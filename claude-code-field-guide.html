01 — Overview

What is Claude Code?

Claude Code is a command-line AI agent that works directly inside your codebase — reading files, writing code, running tests, and making changes autonomously. It is not a chat interface you paste code into.

≠

Not a chatbot

Claude Code operates as an agent in your terminal. It has full access to your file system, can run commands, and executes multi-step tasks without you copy-pasting anything.

↻

Plan → Execute → Verify

Every serious task follows a deliberate rhythm: Claude explores, writes a plan for your approval, then executes — pausing at checkpoints and verifying results before calling anything done.

⊕

You stay in control

Plan files require your approval before a single file changes. You can interrupt at any time, rewind to a prior state, or redirect. Claude doesn't hide what it's doing.

Install: curl -fsSL https://claude.ai/install.sh | bash  ·  then cd your-project && claude

02 — The Mental Model

The Ralph Loop

Claude Code is built around a pattern called the Ralph loop — a simple idea that explains why the tool works the way it does. Understanding this changes how you use it.

Coined by Geoffrey Huntley and popularized when Claude Code's creator Boris Cherny shared the pattern, the Ralph loop describes a way to use AI coding agents that actually works reliably: instead of one long conversation that gradually loses focus, you run repeated short loops — each with fresh context — until the job is done. Memory lives in files, not in the model.

The core insight: AI agents degrade in long conversations. The longer the context grows, the more Claude loses focus and makes mistakes. The Ralph loop fixes this by keeping each iteration short and bounded — all continuity is stored in plan files, task lists, and git history, not in the conversation. A deterministic loop operating in a non-deterministic environment.

03 — Core Mechanics

The Five Primitives

Claude Code gives you five building blocks. Skills and subagents are the most important to understand. The rest work mostly behind the scenes once you know the loop.

1
Skills
tells Claude HOW to work

Skills are named, built-in workflows that override Claude's default approach. Instead of improvising, Claude follows a proven process. Invoke them by name in your prompt — Claude will announce which skill it's using and why before it starts.

Think of skills as the difference between asking a new hire to "fix that bug" versus "follow our structured debugging runbook on that bug." Same request, very different process and reliability.

Skill Use when What it does differently
superpowers:brainstorming Before building anything Explores intent and alternatives before jumping to solutions
superpowers:writing-plans Turning requirements into a plan Produces a structured, bounded plan file with clear acceptance criteria
superpowers:executing-plans Running a plan you've approved Works through plan steps with checkpoints for review between phases
superpowers:systematic-debugging Any bug or unexpected behavior Diagnoses root cause first, then fixes — never guesses first
superpowers:verification-before-completion Before declaring anything done Runs tests and inspects real output — no success without evidence
superpowers:test-driven-development When implementing new features Writes failing tests first, then implements to make them pass
2
Subagents
Agent tool · parallel workers

For larger tasks, Claude spawns specialized child agents — each given exactly the context they need and nothing more. This is the Ralph loop's "allocate context" discipline in action: fresh instances, tight scope.

Multiple subagents can run concurrently for independent sub-problems, which makes execution significantly faster. You don't manage them directly — they appear automatically when you use /plan or superpowers:executing-plans.

Explore agent

Read-only codebase investigation. Understands structure and finds relevant files without touching anything.

Plan agent

Designs the approach for a specific task based on what the Explore agents discovered.

General-purpose agent

Does the actual work: research, implementation, running commands, multi-step changes.

3
Plan Files
~/.claude/plans/ · the contract

When you run /plan, Claude writes a structured Markdown file capturing: what it understands about your goal, what it found in the codebase, the specific steps it will take, and what "done" looks like. Nothing changes until you approve it.

Plan files are also the Ralph loop's persistent memory — they survive across sessions, so you can stop and pick up exactly where you left off. Think of a plan file as a contract: Claude won't change your code until you've signed off.

4
Tasks
TaskCreate · TaskUpdate · TaskList

While executing a plan, Claude maintains a live task checklist. Tasks are created automatically from plan items and updated as Claude works through them. You'll see them referenced in Claude's output as it progresses. Mostly automatic — you observe rather than manage.

5
/loop
autonomous iteration command

The Ralph loop made into a single command. Give Claude a clear goal and let it run autonomously — checking its own work and looping until done. Two modes:

/loop

Self-paced autonomous execution. Claude runs until it produces a completion signal. Best for well-defined tasks with clear acceptance criteria.

/loop 5m /check-deploy

Scheduled recurring loop. Runs the command every 5 minutes. Useful for monitoring, polling, or CI/CD-style automation.

04 — End to End

A Complete Workflow

Here's how a real task moves from idea to shipped — using the loop model and the primitives together.

You

Describe what you want

Write out the feature, bug, or refactor in plain language. The more specific you are — what the code does, what's broken, what "done" looks like — the better Claude's plan will be. Vague prompts produce vague plans.

Command

/plan — Claude enters plan mode

Claude spawns Explore subagents to read the relevant parts of your codebase, then a Plan subagent to design the approach. It will ask clarifying questions before writing anything. The result is a plan file in ~/.claude/plans/ for your review.

You

Read and approve the plan

This takes two minutes and prevents hours of surprises. If something looks off, say so — Claude will revise. No code changes until you approve. This is the loop's "allocate context" step — you're verifying Claude understood the task correctly.

Skill

superpowers:executing-plans takes over

Claude reads the plan file and works through it step by step, spawning parallel subagents for independent tasks. It pauses at review checkpoints between phases. You stay informed without having to babysit every action.

  • Subagents run concurrently where tasks are independent
  • Progress tracked via the live task list
  • Each iteration stays within a bounded context window
Skill

superpowers:verification-before-completion

Before Claude tells you it's done, it runs tests, checks actual output, and confirms results. No success is claimed without real evidence. This is the loop's "verify" step — the most important one most people skip.

Optional

superpowers:requesting-code-review → PR or merge

Claude can open a pull request or hand off for merge once verification passes.

For fully autonomous work: once you trust the plan, swap to /loop and let Claude iterate until done. Review the output afterward. The plan file and git history give you a complete audit trail of what happened.

07 — Getting Started

Your First Four Sessions

Do these in order. Each one builds intuition you'll need for the next.

1

Run /plan on something real

Don't start by asking Claude to write code. Pick a real task — a bug fix, a small feature — and run /plan first. Watch Claude explore your codebase, ask questions, and write a plan. Read it before approving. This builds the core habit.

2

Use a skill explicitly

Find a bug. Instead of "can you look at this bug," say "use superpowers:systematic-debugging to investigate why X is happening." Notice how the output is more structured and methodical. Skills are not just names — they change the process significantly.

3

Run /loop on a bounded task

Take something small and well-defined. Write it up clearly with an explicit success condition ("done when all tests pass"). Run /loop and let Claude iterate. Review the result and the git diff afterward.

4

Create a CLAUDE.md

Run /init in your project. This creates a file Claude reads at the start of every session — your team's coding standards, architecture decisions, things Claude should always know. The more useful this file is, the better every session gets.

Things worth knowing before you start

Context

More specific prompts produce better plans. Include what the code does, what's wrong, and what "done" looks like. Vague in, vague out.

Skills

Naming skills explicitly — not just describing what you want — produces significantly more reliable results. "Use systematic-debugging" ≠ "debug this."

Plans

Always read the plan before approving. This is the cheapest possible time to catch a misunderstanding — before a single file changes.

Tasks

Slice tasks small. "Add a filter to the table" is a good task. "Rebuild the dashboard" is not. Large tasks cause context overflow and poor output.
